# Plans

This directory contains all planning documents for the project.

## Structure

```
docs/plans/
  README.md                    # This file
  YYYY-wNN/                    # Weekly sprint folder (one per week)
    YYYY-wNN-goals.md          # Goals, priorities, demo script
    sprint-plans/              # Implementation plans for this sprint
      P0.1-feat-some-feature.md
      P1.2-fix-some-bug.md
  dev-plans/                   # Standalone strategy & architecture docs
    YYYY-MM-DD-some-topic.md
```

## Weekly sprints (`YYYY-wNN/`)

Each week gets a folder containing:

- **Goals file** (`YYYY-wNN-goals.md`) — North Star, priorities (P0/P1/P2), scope boundaries, success criteria, demo script
- **Sprint plans** (`sprint-plans/`) — detailed implementation plans with file footprints, test criteria, and E2E verification steps

### Plan formats

Sprint plans come in two formats:

**Single-file plans** — for focused, self-contained work:
```
sprint-plans/
  P0.1-feat-some-feature.md
  P1.2-fix-some-bug.md
```

**Folder-based plans** — for complex workstreams with multiple sub-plans, shared architecture context, or parallel execution tracks:
```
sprint-plans/
  my-workstream/                     # Complex workstream folder
    00-architecture-overview.md      # Entry point — architecture context, dependency graph
    G1-first-sub-task.md             # Sub-plan (executable by /build)
    G2-second-sub-task.md            # Sub-plan
    G3-third-sub-task.md             # Sub-plan
```

**Folder conventions:**
- `00-*.md` is always the entry point — architecture context, dependency graph, parallelization strategy
- Sub-plans follow the same template as single-file plans (file footprint, implementation steps, test plan, done criteria)
- Sub-plans are individually executable by `/build <path-to-sub-plan>`
- Completed sub-plans get renamed with `.DONE` suffix
- When ALL sub-plans in a folder are `.DONE`, the workstream is considered complete

**When to use a folder vs. a single file:**
- Single file: one coherent deliverable, ~3-5 hours, no internal parallelism
- Folder: multiple related deliverables that share architectural context, have internal dependencies or parallelization opportunities, or would be >8 hours as a single plan

**Referencing in goals doc:**
```markdown
| # | Project | Plan | Stream | Status |
|---|---------|------|--------|--------|
| 1 | My Feature | [P0.1](sprint-plans/P0.1-feat-my-feature.md) | A | Planned |
| 3 | Big Workstream | [my-workstream/](sprint-plans/my-workstream/00-architecture-overview.md) | B | Planned |
```

### Planning principles

- **One big thing per week.** The North Star should be singular. If you can't say it in one sentence, the week is over-scoped.
- **P0 means 2-3 items, not 5.** If everything is P0, nothing is.
- **Demo scripts have a cut line.** Scenes are tagged `[Exists]`, `[Extend]`, `[Build]`, or `[Narrate]`. Build/extend hours must fit within the week's capacity.
- **Quality over quantity.** A polished 3-minute demo beats a broken 10-minute one.
- **Plans are specs, not wishes.** Every plan must be detailed enough for autonomous execution.

## Dev plans (`dev-plans/`)

Standalone strategy, architecture, and feature planning documents that aren't tied to a specific sprint week.

## Goals document template

```markdown
# Week YYYY-wNN Goals

## North Star

One sentence: what does success look like at the end of this week?

## Priorities

### P0 — Must ship
- [ ] P0.1 [item name] — brief description
- [ ] P0.2 [item name] — brief description

### P1 — Should ship if P0 is done
- [ ] P1.1 [item name] — brief description

### P2 — Nice to have
- [ ] P2.1 [item name] — brief description

## What's NOT This Week

Explicit scope boundaries — things that are tempting but out of scope.
- [thing to defer]

## What Already Works

Context on the current state — don't re-build what exists.
- [capability 1]

## Demo Script

[Generated by /demo-script skill]

## Sprint Plan

[Generated by /plan-sprint skill]

| # | Project | Plan | Stream | Status |
|---|---------|------|--------|--------|
| 1 | [name]  | [link] | A | Planned |
```

## Claude integration

The full workflow from planning to shipping:

```
/weekly-goals  →  /demo-script  →  /plan-sprint  →  /build  →  /sync
   (why)            (what)           (how)          (do it)    (reset)
```

- **`/weekly-goals`** — loads the current week's goals, keeps work aligned, guards capacity
- **`/demo-script`** — generates or updates the demo script, scoped to what can ship this week
- **`/plan-sprint`** — breaks goals into concrete projects with file footprints, test criteria, and E2E verification steps
- **`/build <plan-path>`** — executes a sprint plan autonomously: creates branch, implements, tests, verifies, commits, creates PR
- **`/ship`** — standalone push + PR flow for simpler changes
- **`/sync`** — switch back to main and pull latest
